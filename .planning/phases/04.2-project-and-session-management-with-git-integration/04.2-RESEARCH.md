# Phase 04.2: Project and session management with git integration - Research

**Researched:** 2026-03-27
**Domain:** Electron IPC (RPC groups), SQLite persistence, git CLI operations, React UI (shadcn dialogs/command)
**Confidence:** HIGH

## Summary

This phase replaces the hardcoded `mockProjects` in the sidebar with real project/session data backed by SQLite, adds git integration for branch listing and worktree management, and introduces native directory picker for project registration. The codebase already has well-established patterns for every major component: RPC groups (`ClaudeRpcGroup`, `PersistenceRpcGroup`), database stores (`TabStore`), process spawning (`CommandExecutor`), and atom state management (`Atom.family`, `Atom.keepAlive`).

The primary challenge is orchestration -- connecting five new concerns (Project table, GitRpcGroup, DialogRpcGroup, extended PersistenceRpcGroup, and sidebar atoms) without disrupting the existing chat infrastructure. All building blocks exist; this phase is integration-heavy, not research-heavy.

**Primary recommendation:** Follow the exact patterns established in Phase 04 (RPC groups, handlers, layer composition) and Phase 02 (database stores). The new `GitRpcGroup` should use `CommandExecutor` from `@effect/platform` to spawn `git` commands, mirroring `ClaudeCliLive`'s process spawning pattern but with simpler synchronous-like output (no streaming).

- **D-01:** "+" button on the project group header opens an in-app dialog/modal for session creation (not inline or popover).

- **D-10:** Session creation dialog has: a name input with autocomplete from existing git branches, and a "use worktree" checkbox.

- **D-12:** If a worktree for the entered branch already exists, offer to reuse it ("A worktree for 'branch' already exists. Use it for this session?").
- **D-13:** Worktree base path is a project-level setting, configured on registration or in project settings. Default suggestions: `.claude/worktrees/`, `.worktrees/`. User can override with a custom path.

- **D-16:** Projects are registered via Electron's `dialog.showOpenDialog` native directory picker. The project name is auto-derived from the git repository name (or folder name if not a git repo).
- **D-17:** Immediately after registration, a first session is auto-created on the project's current git branch.
- **D-18:** Projects are a persistent concept -- they remain in the sidebar even if all sessions are closed.
- **D-19:** Remove only -- no rename. Projects use the git repo name as their label.
- **D-20:** Removing a project closes all its sessions and cleans up associated state (atoms, tab persistence).
- **D-21:** New `GitRpcGroup` for git operations: `listBranches`, `getCurrentBranch`, `createWorktree`, `removeWorktree`, `getRepoName`, `listWorktrees`. Uses `CommandExecutor` from `@effect/platform` (same pattern as `ClaudeCliLive`).

- **D-23:** Extend existing `PersistenceRpcGroup` with project CRUD: `createProject`, `listProjects`, `removeProject`. Projects are persistence data stored in SQLite.

### Pattern 1: Git service using CommandExecutor

The `ClaudeCliLive` service already demonstrates spawning CLI processes via `CommandExecutor`. Git operations follow the same pattern but are simpler (non-streaming, single output):

```typescript
// Source: Existing pattern from src/services/claude-cli/service.ts
const runGitCommand = (args: string[], cwd: string) =>
  Effect.gen(function* () {
    let command = Command.make("git", ...args);
    command = Command.workingDirectory(command, cwd);
    const process = yield* Command.start(command).pipe(
      Effect.mapError((cause) =>
        new GitOperationError({ message: String(cause), operation: args[0] })
      ),
    );
    const stdout = yield* process.stdout.pipe(
      Stream.decodeText(),
      Stream.runFold("", (acc, s) => acc + s),
    );
    const exitCode = yield* process.exitCode;
    if (exitCode !== 0) {
      const stderr = yield* process.stderr.pipe(
        Stream.decodeText(),
        Stream.runFold("", (acc, s) => acc + s),
      );
      return yield* new GitOperationError({
        message: stderr.trim(),
        operation: args[0],
      });
    }
    return stdout.trim();
  });
```

### Pattern 4: Electron dialog in Effect layer

Electron's `dialog.showOpenDialog` is async and returns a promise. Wrap in Effect:

```typescript
// DialogServiceLive implementation
import { dialog } from "electron";

const openDirectory = () =>
  Effect.tryPromise({
    try: () =>
      dialog.showOpenDialog({
        properties: ["openDirectory"],
        title: "Select Project Directory",
      }),
    catch: (cause) =>
      new DialogError({ message: String(cause), operation: "openDirectory" }),
  }).pipe(
    Effect.map((result) =>
      result.canceled ? Option.none() : Option.some(result.filePaths[0])
    ),
  );
```

### Pattern 5: Atom cache with DB-first writes (D-06)

```typescript
// projectsAtom: loaded from DB on app start, refreshed after mutations
export const projectsAtom = Atom.make<ReadonlyArray<Project>>([]).pipe(
  Atom.keepAlive,
);

// Load projects from DB via RPC on app start
export const loadProjectsAtom = appRuntime.fn(
  (_: void, ctx: Atom.FnContext) =>
    Effect.gen(function* () {
      const client = yield* RpcClient.make(MergedRpcGroup);
      const projects = yield* client.listProjects({});
      ctx.set(projectsAtom, projects);
    }),
);
```

### Anti-Patterns to Avoid

- **Atom-first mutations:** Never update atoms before the DB write succeeds. Always DB-first, then refresh atom cache (D-06).
- **Importing `electron` in renderer code:** The `dialog` module is main-process only. Always access via RPC.
- **Streaming RPC for git operations:** Git operations return single results, not streams. Use non-streaming `Rpc.make` (no `stream: true`).
- **Hand-rolling branch autocomplete filtering:** Use the shadcn `command` component which has built-in fuzzy filtering via `cmdk`.

## Don't Hand-Roll

- **Branch autocomplete filtering:** Use shadcn `command` component (wraps `cmdk`) which provides fuzzy search, keyboard navigation, and empty state -- all built in.
- **Directory picker dialog:** Use Electron's native `dialog.showOpenDialog` -- OS-native UX, no custom file browser needed.
- **Context menu:** Use shadcn `context-menu` component which handles right-click binding, positioning, keyboard accessibility.
- **Git operations parsing:** `git branch --list --format` and `git worktree list --porcelain` provide structured output. Parse line-by-line, no JSON needed.

## Common Pitfalls

### Pitfall 1: Tabs table needs project_id foreign key
**What goes wrong:** The existing `tabs` table has no `project_id` column. Adding it requires a schema migration.
**Why it happens:** Phase 02 designed tabs before the project concept existed.
**How to avoid:** Add `project_id INTEGER REFERENCES projects(id)` column to the tabs table DDL. Since the app uses `CREATE TABLE IF NOT EXISTS` and no migration framework, either: (a) add the column via `ALTER TABLE` on startup if it doesn't exist, or (b) bump PRAGMA user_version and conditionally run the migration.
**Warning signs:** Foreign key violations or null project_id on existing tabs.

### Pitfall 2: RPC group merge order and handler dependencies
**What goes wrong:** The merged group in `server.ts` and `client.ts` must be identical. If they diverge, RPC calls fail silently or with opaque errors.
**Why it happens:** Two separate files define the merge independently.
**How to avoid:** Define the merged group in a single shared file (e.g., `src/services/rpc-groups.ts`) and import from both server and client.
**Warning signs:** RPC calls returning unexpected errors or hanging.

### Pitfall 3: Electron dialog must run on main process
**What goes wrong:** Calling `dialog.showOpenDialog` from renderer crashes.
**Why it happens:** Electron's `dialog` module is only available in the main process.
**How to avoid:** Always access via `DialogRpcGroup` RPC call from renderer. The handler on main process imports `dialog` from `electron`.
**Warning signs:** `TypeError: dialog.showOpenDialog is not a function` in renderer console.

### Pitfall 4: Git worktree path conflicts
**What goes wrong:** `git worktree add` fails if the target directory already exists or if the branch already has a worktree.
**Why it happens:** User enters a branch that already has a worktree elsewhere.
**How to avoid:** Before calling `git worktree add`, call `git worktree list` and check. Implement D-12 (reuse prompt) based on the list result.
**Warning signs:** Git error: "fatal: '{path}' is already a worktree".

### Pitfall 5: SQLite boolean storage
**What goes wrong:** SQLite has no native boolean type. `is_git_repo` stored as INTEGER (0/1) may decode incorrectly.
**Why it happens:** Effect Schema expects boolean but SQLite returns 0/1.
**How to avoid:** Use `Schema.transform` to map 0/1 to false/true, or store as INTEGER and transform in the decode function.
**Warning signs:** Schema decode errors on project load.

### Pitfall 6: CASCADE delete vs manual cascade
**What goes wrong:** SQLite foreign key enforcement is off by default.
**Why it happens:** `PRAGMA foreign_keys = ON` must be set per connection.
**How to avoid:** Either enable the pragma (check if existing Database layer does this) or do manual cascade in a transaction (same pattern as TabStore's delete).
**Warning signs:** Orphaned tab rows after project removal.

    cwd,
  ).pipe(Effect.map(() => worktreePath));

## Open Questions

1. **Schema migration strategy for adding project_id to tabs**
   - What we know: The app uses `CREATE TABLE IF NOT EXISTS` with no migration framework. PRAGMA user_version is available but not currently used.
   - What's unclear: Whether existing tabs should be assigned to a "default" project or left with null project_id.
   - Recommendation: Use `ALTER TABLE tabs ADD COLUMN project_id INTEGER REFERENCES projects(id)` guarded by a check (try/catch on the ALTER). Existing tabs get null project_id. New tabs always have project_id.

2. **Atom cache structure for projects**
   - What we know: `Atom.family` works well for per-entity state (used for chat atoms). A single `projectsAtom` with derived selectors is simpler for a list.
   - What's unclear: Whether a flat list atom or `Atom.family` per project is better for sidebar reactivity.
   - Recommendation: Use a flat `projectsAtom: Atom<ReadonlyArray<ProjectWithSessions>>` since the sidebar renders all projects at once. Individual project state doesn't need independent subscriptions.

3. **Git worktree with existing branch vs new branch**
   - What we know: `git worktree add <path> <branch>` checks out an existing branch. `git worktree add <path> -b <branch>` creates a new branch.
   - What's unclear: Whether the session creation dialog should differentiate between "checkout existing branch in worktree" and "create new branch in worktree".
   - Recommendation: If the branch exists, use `git worktree add <path> <branch>`. If it doesn't exist, use `git worktree add <path> -b <branch>`. Detect automatically by checking `git branch --list <name>`.
