# Phase 04.2: Project and session management with git integration - Context

**Gathered:** 2026-03-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Project registration and session creation flows within the tab sidebar. Users register projects via a native directory picker, create sessions within projects by specifying a branch name and choosing branch vs worktree, and remove projects (cascading to all sessions). The app manages git worktree creation via `git worktree add`. Requires new RPC endpoints on the main process for git operations and native dialogs. Does not include multi-stream orchestration or background streaming — that's Phase 04.3.

</domain>

<decisions>
## Implementation Decisions

### Sidebar interactions
- **D-01:** "+" button on the project group header opens an in-app dialog/modal for session creation (not inline or popover).
- **D-02:** Project removal via right-click context menu on the project header.
- **D-03:** Always show a confirmation dialog (in-app modal) before removing a project, displaying how many sessions will be closed.
- **D-04:** Clicking a session in the sidebar shows a brief loading transition (skeleton) while the session's atom state is resolved, then renders the chat.
- **D-05:** Session creation dialog is also an in-app modal (consistent with removal confirmation).

### Project/session data model
- **D-06:** Database-first with atom cache. Projects live in SQLite. On app start, load from DB into atoms. Mutations go to DB first, then refresh atoms.
- **D-07:** New `Project` table in SQLite (id, name, directory, is_git_repo, worktree_base_path). Sessions (tabs) reference `project_id` via foreign key. Separate from the existing Tab table.
- **D-08:** Eager load all projects and their sessions from DB into atoms on app start, before rendering the sidebar.
- **D-09:** When a new project is registered, it auto-expands in the sidebar and its auto-created first session becomes the active tab.

### Session creation flow
- **D-10:** Session creation dialog has: a name input with autocomplete from existing git branches, and a "use worktree" checkbox.
- **D-11:** "Use worktree" checkbox is checked by default. When checked, `git worktree add` is called with the provided name. When unchecked, just switch/create a branch.
- **D-12:** If a worktree for the entered branch already exists, offer to reuse it ("A worktree for 'branch' already exists. Use it for this session?").

### Worktree management
- **D-13:** Worktree base path is a project-level setting, configured on registration or in project settings. Default suggestions: `.claude/worktrees/`, `.worktrees/`. User can override with a custom path.
- **D-14:** When a session with a worktree is removed, ask the user: "Also remove the worktree at [path]?" — user decides each time.
- **D-15:** Non-git directories are allowed for project registration but git features (worktree checkbox, branch autocomplete) are disabled. Session just uses the directory as cwd.

### Project registration
- **D-16:** Projects are registered via Electron's `dialog.showOpenDialog` native directory picker. The project name is auto-derived from the git repository name (or folder name if not a git repo).
- **D-17:** Immediately after registration, a first session is auto-created on the project's current git branch.
- **D-18:** Projects are a persistent concept — they remain in the sidebar even if all sessions are closed.

### Project management
- **D-19:** Remove only — no rename. Projects use the git repo name as their label.
- **D-20:** Removing a project closes all its sessions and cleans up associated state (atoms, tab persistence).

### RPC architecture
- **D-21:** New `GitRpcGroup` for git operations: `listBranches`, `getCurrentBranch`, `createWorktree`, `removeWorktree`, `getRepoName`, `listWorktrees`. Uses `CommandExecutor` from `@effect/platform` (same pattern as `ClaudeCliLive`).
- **D-22:** New `DialogRpcGroup` for native Electron dialogs: `openDirectory`. Clean separation — reusable beyond project registration.
- **D-23:** Extend existing `PersistenceRpcGroup` with project CRUD: `createProject`, `listProjects`, `removeProject`. Projects are persistence data stored in SQLite.
- **D-24:** All three RPC groups merged via `RpcGroup.merge` into the combined group (same pattern as Phase 04).

### Claude's Discretion
- GitRpcGroup handler implementation details (error mapping, git command arguments)
- DialogRpcGroup handler implementation (how to access Electron's `dialog` API from within Effect layers)
- Project table schema details (column types, constraints, indices)
- In-app modal component design (which shadcn primitives to compose)
- Branch autocomplete implementation (debouncing, filtering, UI component)
- Loading transition skeleton design for session switching
- Context menu implementation for project removal (shadcn dropdown-menu or custom)
- Atom cache structure (single projectsAtom with derived selectors, or atom family)

</decisions>

### Established Patterns
- **RPC group definition**: `RpcGroup.make(Rpc.make(...))` in `group.ts` — follow for git and dialog operations
- **Process spawning**: `CommandExecutor` + `Command.make("git")` for git CLI operations
- **Layer composition**: `Layer.provideMerge` chain in `main.ts` — new service layers slot in here
- **Database services**: ProjectStore follows same pattern as TabStore (service-definition.ts + service.ts + schemas.ts)
- **Atom state**: `Atom.make()` with `Atom.keepAlive` for persistent state, `Atom.family()` for per-entity state

### Integration Points

- `src/services/persistence-rpc/group.ts` — Add project CRUD RPCs
- `src/services/persistence-rpc/handlers.ts` — Add project CRUD handlers

<specifics>
## Specific Ideas

- Native directory picker for project registration — familiar OS-level UX
- Git worktree creation managed by the app — user just picks a branch name and type
- First session auto-created on registration to minimize friction
- Worktree base path as a project-level setting with sensible defaults (.claude/worktrees/, .worktrees/)
- Branch autocomplete in session creation dialog for discoverability

</specifics>

<deferred>
## Deferred Ideas

- Project renaming — use git repo name only for now
- Session renaming/relabeling — use branch name only for now
- Worktree cleanup automation (auto-remove on session close without asking) — deferred; currently asks user each time
- Project settings UI (beyond worktree path) — future phase if needed
- Branch creation without worktree isolation — supported but not the default flow

</deferred>
