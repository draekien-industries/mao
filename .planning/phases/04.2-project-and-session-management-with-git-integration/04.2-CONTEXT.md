# Phase 04.2: Project and session management with git integration - Context

**Gathered:** 2026-03-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Project registration and session creation flows within the tab sidebar. Users register projects via a native directory picker, create sessions within projects by specifying a branch name and choosing branch vs worktree, and remove projects (cascading to all sessions). The app manages git worktree creation via `git worktree add`. Requires new RPC endpoints on the main process for git operations and native dialogs. Does not include multi-stream orchestration or background streaming — that's Phase 04.3.

</domain>

<decisions>
## Implementation Decisions

### Project registration
- **D-01:** Projects are registered via Electron's `dialog.showOpenDialog` native directory picker. The project name is auto-derived from the git repository name (or folder name if not a git repo).
- **D-02:** Immediately after registration, a first session is auto-created on the project's current git branch. One click to register, ready to chat.
- **D-03:** Projects are a persistent concept — they remain in the sidebar even if all sessions are closed.

### Project management
- **D-04:** Remove only — no rename. Projects use the git repo name as their label.
- **D-05:** Removing a project closes all its sessions and cleans up associated state (atoms, tab persistence).

### Session creation
- **D-06:** "+" button on the project group header in the sidebar creates a new session.
- **D-07:** If no sessions exist for the project, the first session defaults to the current git branch name as its label.
- **D-08:** If sessions already exist, the user is prompted to provide a branch name and choose whether to create a branch or a worktree.
- **D-09:** When the user chooses "worktree", the app runs `git worktree add` to create it. The app manages worktree creation — no manual setup required.
- **D-10:** Session labels display the branch/worktree name.

### Git operations
- **D-11:** New RPC endpoints needed on the main process for: detecting git repo name from a directory, listing branches, getting current branch, creating worktrees. These are main-process operations because the renderer cannot access the filesystem directly.
- **D-12:** Git operations use Effect's `CommandExecutor` from `@effect/platform` (same pattern as `ClaudeCliLive` for spawning `claude` CLI).

### Claude's Discretion
- Git RPC group design (extend PersistenceRpcGroup or create a new GitRpcGroup)
- Native dialog RPC design (how to expose `dialog.showOpenDialog` to the renderer)
- Branch/worktree picker UI component design
- Project persistence strategy (localStorage, database, or atoms with persistence)
- Error handling for git operations (non-git directories, worktree conflicts, etc.)
- Worktree directory naming and location conventions

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `CommandExecutor` from `@effect/platform` — Already used by `ClaudeCliLive` for spawning processes; reuse for `git` commands
- `ElectronServerProtocol` / `ElectronClientProtocol` — Existing IPC transport for new RPC endpoints
- `PersistenceRpcGroup` — May be extended with git/dialog operations, or a new group created alongside it
- Sidebar component structure from Phase 04.1 — "+" button placement, project group headers

### Established Patterns
- **RPC group definition**: `RpcGroup.make(Rpc.make(...))` in `group.ts` — follow for git operations
- **Process spawning**: `CommandExecutor` + `Command.make("git")` for git CLI operations
- **Layer composition**: `Layer.provideMerge` chain in `main.ts` — new git service layers slot in here

### Integration Points
- `src/main.ts` — New git service layers and RPC handlers need to be wired into the layer composition
- `src/preload.ts` — May need Electron `dialog` API exposure if not routed through RPC
- Sidebar component — "+" button triggers session creation flow, project header has remove action

</code_context>
