# Phase 04.5: Fix tab working directory mismatch - Context

**Gathered:** 2026-03-28
**Status:** Ready for planning

<domain>
## Phase Boundary

Ensure each tab's Claude CLI process spawns in the correct project directory. Currently `sendMessageAtom` creates `QueryParams`/`ResumeParams` without passing the tab's `cwd`, so the CLI runs in the app's default directory instead of the registered project directory. This phase threads the tab's stored `cwd` through to CLI process spawning for both new queries and resumed sessions.

</domain>

<decisions>
## Implementation Decisions

### CWD Threading
- **D-01:** Add `cwdAtom = Atom.family(tabId => Atom.make<string>(""))` in `src/atoms/chat.ts`, following the existing per-tab family pattern with `Atom.keepAlive`
- **D-02:** `sendMessageAtom` reads `ctx(cwdAtom(tabId))` and passes `cwd` to both `new QueryParams({ prompt, cwd })` and `new ResumeParams({ prompt, session_id, cwd })`
- **D-03:** `cwdAtom` is populated on tab activation — when `setActiveTabAtom` fires, it looks up the tab's `cwd` from `projectsAtom` and sets `cwdAtom(tabId)`

### Tab Metadata Access
- **D-04:** `chat.ts` already imports `activeTabIdAtom` from `sidebar.ts` — no additional cross-module imports needed
- **D-05:** `setActiveTabAtom` (in `sidebar.ts`) imports and populates `cwdAtom` (in `chat.ts`) during tab switch. Sidebar writes, chat reads — minimal coupling
- **D-06:** No new tab metadata atoms beyond `cwdAtom` — other metadata (branch, project) stays in `projectsAtom`

### Resume Behavior
- **D-07:** Always pass `cwd` for both `QueryParams` and `ResumeParams`. The `--resume` flag tells Claude which conversation to continue, but the spawned process still needs the correct working directory for file operations, tool use, and context
- **D-08:** Consistent behavior — no special-casing for resume vs. new query regarding cwd

### Claude's Discretion
- How to handle the edge case where `cwdAtom(tabId)` is empty (e.g., tab activated before projects load) — likely a guard or fallback
- Whether `registerProjectAtom` and `createSessionAtom` should also eagerly set `cwdAtom` for the newly created tab (in addition to `setActiveTabAtom`)

</decisions>
