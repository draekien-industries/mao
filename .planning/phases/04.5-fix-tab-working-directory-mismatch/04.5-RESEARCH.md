# Phase 04.5: Fix tab working directory mismatch - Research

**Researched:** 2026-03-28
**Domain:** Atom state threading / CLI process spawning
**Confidence:** HIGH

## Summary

This phase fixes a bug where `sendMessageAtom` constructs `QueryParams` and `ResumeParams` without passing the tab's `cwd`, causing every Claude CLI process to spawn in the app's default directory instead of the project directory associated with the tab. The fix is narrow: add a `cwdAtom` family to `chat.ts`, populate it when tabs activate, and thread the value into `QueryParams`/`ResumeParams` construction in `sendMessageAtom`.

The downstream plumbing already exists. `QueryParams` and `ResumeParams` both accept an optional `cwd` field (via `sharedSchemaFields`), and `buildStream` in `service.ts` already applies `Command.workingDirectory(command, params.cwd)` when `cwd` is truthy. No schema, service, or RPC changes are needed.

**Primary recommendation:** Add `cwdAtom` family in `chat.ts`, populate it from `projectsAtom` on tab switch in `sidebar.ts`, and pass `ctx(cwdAtom(tabId))` into both `QueryParams` and `ResumeParams` constructors in `sendMessageAtom`.

### Pattern: cwdAtom family (follows existing conventions)

All per-tab atom families in `chat.ts` use the same shape:

```typescript
export const cwdAtom = Atom.family((_tabId: string) =>
  Atom.make("").pipe(Atom.keepAlive),
);
```

This follows the exact pattern of `messagesAtom`, `streamingTextAtom`, `sessionIdAtom`, `isStreamingAtom`, `errorAtom`, `eventsAtom`, `unreadAtom`, `toolInputAtom`, and `draftInputAtom`.

### Pattern: Cross-module atom population (sidebar writes, chat reads)

`setActiveTabAtom` in `sidebar.ts` already imports from `chat.ts` (the `unreadAtom` is cleared on tab switch currently -- no, actually `unreadAtom` is cleared by scroll, not setActiveTab). This cross-module write is the established coupling direction. Sidebar imports `cwdAtom` from `chat.ts` and writes to it.

### Pattern: Tab cwd lookup from projectsAtom

`projectsAtom` holds `ReadonlyArray<ProjectWithSessions>` where each `ProjectWithSessions` has `sessions: ReadonlyArray<Tab>`. Each `Tab` has a `cwd: string` field. The lookup:

```typescript
const projects = ctx(projectsAtom);
const tab = projects
  .flatMap((p) => p.sessions)
  .find((s) => s.id === tabId);
if (tab) {
  ctx.set(cwdAtom(String(tabId)), tab.cwd);
}
```

### Pattern: sendMessageAtom cwd threading

Lines 126-133 of `chat.ts` construct params without `cwd`. The fix adds one read:

```typescript
const cwd = ctx(cwdAtom(tabId));

const stream = currentSessionId
  ? cli.resume(new ResumeParams({ prompt, session_id: currentSessionId, cwd: cwd || undefined }))
  : cli.query(new QueryParams({ prompt, cwd: cwd || undefined }));
```

`cwd` is `Schema.optional(Schema.String)` in both param classes, so passing `undefined` when empty is safe and means no `Command.workingDirectory` is applied (existing fallback behavior).

### Anti-Patterns to Avoid
- **Storing cwd in sendMessageAtom params:** The caller (UI component) should not need to know or pass cwd. The atom layer resolves it from tab state.
- **Reading projectsAtom inside sendMessageAtom:** Violates the separation. `cwdAtom` is the cache; sidebar populates it, chat reads it.

## Common Pitfalls

### Pitfall 1: Empty cwd on first tab activation at app start
**What goes wrong:** `loadProjectsAtom` sets `activeTabIdAtom` directly (line 60) without going through `setActiveTabAtom`, so `cwdAtom` would not be populated for the initial tab.
**Why it happens:** `loadProjectsAtom` uses `ctx.set(activeTabIdAtom, firstTab.id)` directly instead of dispatching `setActiveTabAtom`.
**How to avoid:** Either (a) also populate `cwdAtom` in `loadProjectsAtom` when setting the initial active tab, or (b) add a guard in `sendMessageAtom` that falls back to looking up cwd from `projectsAtom` when `cwdAtom` is empty. Recommendation: option (a) -- populate eagerly in `loadProjectsAtom` since the tab data is already loaded at that point.
**Warning signs:** First message after app launch spawns CLI in wrong directory; subsequent tab switches work correctly.

### Pitfall 2: registerProjectAtom and createSessionAtom bypass setActiveTabAtom
**What goes wrong:** Both `registerProjectAtom` (line 155) and `createSessionAtom` (line 222) set `activeTabIdAtom` directly, bypassing `setActiveTabAtom`. New tabs created via these paths won't have `cwdAtom` populated.
**Why it happens:** These atoms set `activeTabIdAtom` directly after creating the tab in the DB.
**How to avoid:** After `ctx.set(activeTabIdAtom, tab.id)` in both atoms, also set `ctx.set(cwdAtom(String(tab.id)), sessionCwd)` -- the cwd is already known at that point (`directory` in registerProjectAtom, `sessionCwd` in createSessionAtom).
**Warning signs:** New project registration or session creation results in CLI spawning in wrong directory on first message.

### Pitfall 3: Tab ID type mismatch (number vs string)
**What goes wrong:** `activeTabIdAtom` stores `number | null`, but atom families use `string` keys. Forgetting `String(tabId)` causes a family key mismatch.
**Why it happens:** SQLite IDs are integers; atom families require string keys.
**How to avoid:** Always use `String(tabId)` when calling `cwdAtom(String(tabId))` in sidebar.ts. The atoms rules file confirms: "convert numeric tab IDs via `String(tabId)` at the call site."

### Pitfall 4: Circular import between chat.ts and sidebar.ts
**What goes wrong:** `sidebar.ts` needs to import `cwdAtom` from `chat.ts`. `chat.ts` already imports `activeTabIdAtom` from `sidebar.ts`. This creates a bidirectional import.
**Why it happens:** Both modules need to share atom state.
**How to avoid:** Verify that the circular import is safe at the module level. Both files export atoms (values), not side effects. ES module circular imports work when both sides export bindings that are initialized before being accessed. Since atom creation is synchronous and happens at module scope, this should be safe. If not, extract `cwdAtom` to a shared module (e.g., `src/atoms/shared.ts`), but try direct import first.

## Code Examples

### cwdAtom definition (chat.ts)

```typescript
// Per-tab working directory, populated by setActiveTabAtom on tab switch
export const cwdAtom = Atom.family((_tabId: string) =>
  Atom.make("").pipe(Atom.keepAlive),
);
```

### setActiveTabAtom population (sidebar.ts)

```typescript
import { cwdAtom } from "@/atoms/chat";

export const setActiveTabAtom = appRuntime.fn(
  (tabId: number, ctx: Atom.FnContext) =>
    Effect.sync(() => {
      ctx.set(sessionLoadingAtom, true);
      ctx.set(activeTabIdAtom, tabId);

      // Populate cwdAtom from projectsAtom for the activated tab
      const projects = ctx(projectsAtom);
      const tab = projects
        .flatMap((p) => p.sessions)
        .find((s) => s.id === tabId);
      if (tab) {
        ctx.set(cwdAtom(String(tabId)), tab.cwd);
      }

      ctx.set(sessionLoadingAtom, false);
    }),
);
```

### sendMessageAtom cwd threading (chat.ts)

```typescript
const cwd = ctx(cwdAtom(tabId));

const stream = currentSessionId
  ? cli.resume(
      new ResumeParams({
        prompt,
        session_id: currentSessionId,
        cwd: cwd || undefined,
      }),
    )
  : cli.query(new QueryParams({ prompt, cwd: cwd || undefined }));
```
