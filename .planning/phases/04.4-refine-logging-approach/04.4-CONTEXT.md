# Phase 04.4: Refine Logging Approach - Context

**Gathered:** 2026-03-28
**Status:** Ready for planning

<domain>
## Phase Boundary

Refine the logging infrastructure and patterns across the Mao Electron app. The quick task (260328-dct) already added E2E structured Effect logging to all services and atoms. This phase improves the overall approach: production error logging to file, log level granularity, devLog consolidation, and consistent conventions.

</domain>

<decisions>
## Implementation Decisions

### Production Logging Strategy
- **D-01:** Production builds log errors only (Effect.logError level) to a file at `app.getPath('logs')`
- **D-02:** Production error logs use structured JSON format (timestamp, level, message, annotations)
- **D-03:** Log file has simple max-size truncation (e.g., 5MB cap, truncate oldest half when exceeded)
- **D-04:** Log file path is always `app.getPath('logs')` — not configurable

### Production Logger Implementation
- **D-05:** Custom file logger built with `Logger.make` API — filters to Error+ level, appends JSON lines to file
- **D-06:** Production file logger exported as a constant (`ProdFileLogger`) from `diagnostics.ts` alongside existing `DevLogger`/`ProdLogger`
- **D-07:** Compose via `Layer.provide` in `main.ts` — no new Effect service definition needed

### devLog Consolidation
- **D-08:** Keep `devLog` helper for lifecycle logging in `main.ts` — these logs bracket runtime creation/disposal where Effect logger is unavailable
- **D-09:** Move `devLog` helper from `main.ts` to `diagnostics.ts` as an exported function for potential reuse by other non-Effect code
- **D-10:** Keep `[mao:lifecycle]` prefix and `!app.isPackaged` guard

### Log Granularity & Levels
- **D-11:** Use three Effect log levels: Debug (per-operation entry/exit in services), Info (lifecycle events, layer construction, RPC server start), Error (all failures)
- **D-12:** Same Debug/Info/Error split applies to both main process services and renderer-side atoms/RPC client
- **D-13:** DevLogger shows all log levels in dev — no minimum level filter
- **D-14:** Migrate existing per-operation `Effect.logInfo` calls to `Effect.logDebug` where they represent method entry/exit

### Log Output Destinations
- **D-15:** Dev builds: console only (stdout for main, DevTools for renderer). No file logging in dev.
- **D-16:** Prod builds: error-only file logger (no console output, since `ProdLogger` = `Logger.none` for non-error levels)

### Claude's Discretion
- Exact max file size for log truncation (5MB suggested, Claude can adjust)
- Truncation implementation details (half-file truncation vs. other strategy)
- Whether `ProdLogger` constant changes or a new `ProdFileLogger` is composed alongside it

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `diagnostics.ts` — Already exports `annotations`, `DevLogger`, `ProdLogger`. New `ProdFileLogger` and `devLog` will live here.
- `Logger.make` from Effect — API for building custom loggers with full control over output
- `Logger.pretty` — Currently used as DevLogger, shows all levels

### Established Patterns
- All services annotate with `Effect.annotateLogs(annotations.service, "name")` and `Effect.annotateLogs(annotations.operation, "method")`
- `Effect.tapError` before `Effect.mapError` on all mutating operations
- Layer construction logs at service init time
- Renderer runtime wires DevLogger via `Layer.provide(DevLogger)`

### Integration Points
- `main.ts` line 66-67: `Layer.provide(app.isPackaged ? ProdLogger : DevLogger)` — production logger swap point
- `src/atoms/runtime.ts` — Renderer runtime layer composition (needs same logger treatment for consistency)
- 16 files across `src/services/` and `src/atoms/` have Effect.log calls that need level migration (Info -> Debug for per-operation logs)

</code_context>

<specifics>
## Specific Ideas

- The production file logger needs to handle Electron's app lifecycle — file handle must be safe to use during `before-quit` cleanup
- `devLog` should accept `app.isPackaged` as a parameter or closure so it works correctly when imported from `diagnostics.ts` (which doesn't import Electron)
- Log level migration is a bulk change across ~16 files — methodical find-and-replace of per-operation `logInfo` to `logDebug`

</specifics>
