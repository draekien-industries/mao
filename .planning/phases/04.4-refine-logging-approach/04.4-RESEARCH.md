# Phase 04.4: Refine Logging Approach - Research

**Researched:** 2026-03-28
**Domain:** Effect-TS logging, Electron file I/O, structured logging patterns
**Confidence:** HIGH

## Summary

This phase refines the logging infrastructure after the E2E logging quick task (260328-dct) added structured Effect logging to all services and atoms. The work falls into four areas: (1) building a production file logger that writes JSON lines to `app.getPath('logs')`, (2) migrating per-operation `logInfo` calls to `logDebug` for proper granularity, (3) consolidating the `devLog` helper into `diagnostics.ts`, and (4) wiring the production file logger into the main process runtime.

The Effect `Logger` module provides everything needed. `Logger.make` gives full control over output formatting, `Logger.filterLogLevel` restricts which levels pass through, and `Logger.structuredLogger` produces a structured object that can be serialized to JSON lines. The `Logger.add` function composes additional loggers alongside the default, which is the cleanest way to add file logging without replacing the existing production logger.

**Primary recommendation:** Use `Logger.structuredLogger` piped through `Logger.map` to JSON.stringify, filtered to Error+ level, with synchronous `fs.appendFileSync` for writes. No async complexity needed for error-only logging.

## Standard Stack

### Core
- **effect** 3.21.0 -- `Logger`, `LogLevel`, `Layer` modules for custom logger construction and composition
- **Node.js fs** (built-in) -- `appendFileSync`, `statSync`, `writeFileSync`, `readFileSync` for log file operations
- **Electron app** (built-in) -- `app.getPath('logs')` for platform-correct log directory

### Key APIs

- `Logger.make(options => void)` -- Create a custom logger with full control over output
- `Logger.structuredLogger` -- Produces `{ logLevel, fiberId, timestamp, message, cause, annotations, spans }` objects
- `Logger.filterLogLevel(logger, predicate)` -- Filter logger to only process certain log levels
- `Logger.replace(Logger.defaultLogger, customLogger)` -- Replace the default logger
- `Logger.add(logger)` -- Add an additional logger alongside the default (returns a Layer)
- `LogLevel.greaterThanEqual(level, LogLevel.Error)` -- Predicate for Error+ filtering

### Alternatives Considered
- **winston / pino:** Overkill for error-only file logging in Electron. Effect's Logger module covers the need.
- **electron-log:** Popular but adds a dependency for something achievable with 20 lines of Effect Logger + fs.
- **Logger.batched:** Considered for buffered writes, but error-only logging is low-volume enough that sync append is simpler and safer (no lost logs on crash).

## Architecture Patterns

### Logger Composition in diagnostics.ts

The `diagnostics.ts` file is the single source of truth for all logging configuration. After this phase it exports:

- `annotations` -- Annotation key constants (unchanged)
- `DevLogger` -- `Logger.pretty` for dev builds (unchanged)
- `ProdLogger` -- `Logger.replace(Logger.defaultLogger, Logger.none)` for silencing console in prod (unchanged)
- `ProdFileLogger` -- New: `Logger.add(fileLogger)` layer that appends JSON lines to file
- `devLog` -- New export: lifecycle logger function (moved from main.ts)

### Pattern 1: Custom File Logger with Logger.make

```typescript
// Source: Effect Logger.d.ts type definitions (verified in node_modules)
import { appendFileSync, statSync, writeFileSync, readFileSync } from "node:fs";
import { Logger, LogLevel } from "effect";

const makeFileLogger = (logFilePath: string, maxBytes: number) =>
  Logger.make<unknown, void>(({ logLevel, message, date, annotations, cause }) => {
    if (!LogLevel.greaterThanEqual(logLevel, LogLevel.Error)) return;

    const entry = JSON.stringify({
      timestamp: date.toISOString(),
      level: logLevel._tag,
      message: String(message),
      annotations: Object.fromEntries(annotations),
      ...(cause._tag !== "Empty" ? { cause: String(cause) } : {}),
    });

    truncateIfNeeded(logFilePath, maxBytes);
    appendFileSync(logFilePath, entry + "\n", "utf-8");
  });
```

### Pattern 2: Log File Truncation

```typescript
const truncateIfNeeded = (filePath: string, maxBytes: number) => {
  try {
    const stat = statSync(filePath);
    if (stat.size > maxBytes) {
      const content = readFileSync(filePath, "utf-8");
      const lines = content.split("\n");
      const halfIndex = Math.floor(lines.length / 2);
      writeFileSync(filePath, lines.slice(halfIndex).join("\n"), "utf-8");
    }
  } catch {
    // File doesn't exist yet -- nothing to truncate
  }
};
```

### Pattern 3: devLog as Exported Function

```typescript
// diagnostics.ts
export const devLog = (msg: string, isPackaged: boolean) => {
  if (!isPackaged) console.log(`[mao:lifecycle] ${msg}`);
};

// main.ts
import { devLog } from "./services/diagnostics";
devLog("app ready", app.isPackaged);
```

The `isPackaged` parameter avoids importing Electron in `diagnostics.ts`, keeping it a pure utility module.

### Pattern 4: Wiring in main.ts

```typescript
// main.ts
import { ProdFileLogger } from "./services/diagnostics";

const logFilePath = path.join(app.getPath("logs"), "mao-errors.log");

const ServerLayer = BaseLayer.pipe(
  Layer.provide(app.isPackaged ? ProdLogger : DevLogger),
  Layer.provide(app.isPackaged ? ProdFileLogger(logFilePath) : Layer.empty),
);
```

Note: `ProdFileLogger` can be a function that takes the file path and returns a Layer via `Logger.add`, or the path can be passed at construction time. This keeps Electron's `app` import out of `diagnostics.ts`.

### Anti-Patterns to Avoid
- **Async file writes in logger:** Logger.make callback is synchronous. Using async writes risks lost error logs on crash. Use `appendFileSync`.
- **Importing Electron in diagnostics.ts:** The diagnostics module should remain pure (no Electron dependency). Pass `app.getPath('logs')` and `app.isPackaged` from `main.ts`.
- **Creating a new Effect Service for logging:** Per D-07, no new service definition needed. Logger composition via Layer is sufficient.
- **Filtering at the log call site:** Don't add `if (isPackaged)` guards around log calls. Let the Logger layer handle level filtering.

## Don't Hand-Roll

- **JSON formatting for structured logs:** Use `Logger.structuredLogger` output shape as reference, but `JSON.stringify` of a plain object is fine for file output. Don't build a custom JSON serializer.
- **Log rotation:** Don't build proper log rotation (logrotate-style). Simple max-size truncation per D-03 is sufficient for a desktop app.
- **Cross-process log aggregation:** Renderer logs stay in DevTools console. Don't try to route renderer logs to the main process file logger.

## Common Pitfalls

### Pitfall 1: HashMap.HashMap annotations are not plain objects
**What goes wrong:** `Logger.Options.annotations` is `HashMap.HashMap<string, unknown>`, not a plain JS object. Calling `JSON.stringify` directly on it produces `{}`.
**Why it happens:** Effect uses immutable persistent data structures internally.
**How to avoid:** Convert with `HashMap.toEntries` or spread into `Object.fromEntries(annotations)` (HashMap implements Iterable of entries).
**Warning signs:** Empty `annotations: {}` in log file output.

### Pitfall 2: Logger.make message is not always a string
**What goes wrong:** The `message` field in Logger.Options is typed as `unknown` (generic `Message`). It may be an array of segments.
**Why it happens:** Effect log functions can accept multiple arguments that get collected.
**How to avoid:** Use `String(message)` or handle array case: `Array.isArray(message) ? message.join(" ") : String(message)`.
**Warning signs:** Log file shows `[object Object]` in message field.

### Pitfall 3: File handle during Electron quit
**What goes wrong:** Writes fail during `before-quit` because the runtime is disposing.
**Why it happens:** The `before-quit` handler disposes the ManagedRuntime, which tears down layers. But `devLog` lifecycle logs and any final error logs need to write.
**How to avoid:** Use `appendFileSync` (synchronous, no handle management). The file logger doesn't hold an open file descriptor -- it opens, writes, and closes on each log entry. This is safe during quit.
**Warning signs:** Missing error entries that should have been logged during shutdown.

### Pitfall 4: Bulk logInfo-to-logDebug migration breaking tests
**What goes wrong:** Tests that assert on log output may break if they depend on specific log levels.
**Why it happens:** Changing `Effect.logInfo` to `Effect.logDebug` alters what appears in test output.
**How to avoid:** Check existing tests for log-level assertions before migrating. Current test files don't appear to assert on log output, but verify.
**Warning signs:** Test failures after level migration.

### Pitfall 5: Cause serialization
**What goes wrong:** `cause` in Logger.Options is `Cause.Cause<unknown>`, a rich data structure. Naive `JSON.stringify` may produce very large or circular output.
**Why it happens:** Effect Cause can contain stack traces, parallel/sequential composition, etc.
**How to avoid:** Use `Cause.pretty(cause)` for human-readable output, or check `cause._tag !== "Empty"` to skip empty causes.
**Warning signs:** Enormous or unreadable cause fields in log entries.

## Code Examples

### Complete ProdFileLogger Implementation

```typescript
// src/services/diagnostics.ts
import { appendFileSync, readFileSync, statSync, writeFileSync } from "node:fs";
import { Logger, LogLevel } from "effect";
import type { HashMap } from "effect";

const MAX_LOG_BYTES = 5 * 1024 * 1024; // 5MB

const truncateIfNeeded = (filePath: string, maxBytes: number) => {
  try {
    const stat = statSync(filePath);
    if (stat.size > maxBytes) {
      const content = readFileSync(filePath, "utf-8");
      const lines = content.split("\n");
      const halfIndex = Math.floor(lines.length / 2);
      writeFileSync(filePath, lines.slice(halfIndex).join("\n"), "utf-8");
    }
  } catch {
    // File doesn't exist yet
  }
};

const annotationsToRecord = (
  annotations: HashMap.HashMap<string, unknown>,
): Record<string, unknown> => {
  const result: Record<string, unknown> = {};
  for (const [key, value] of annotations) {
    result[key] = value;
  }
  return result;
};

export const makeProdFileLogger = (logFilePath: string) => {
  const fileLogger = Logger.make<unknown, void>(
    ({ logLevel, message, date, annotations, cause }) => {
      if (!LogLevel.greaterThanEqual(logLevel, LogLevel.Error)) return;

      const entry = JSON.stringify({
        timestamp: date.toISOString(),
        level: logLevel._tag,
        message: Array.isArray(message) ? message.join(" ") : String(message),
        annotations: annotationsToRecord(annotations),
        ...(cause._tag !== "Empty"
          ? { cause: String(cause) }
          : {}),
      });

      truncateIfNeeded(logFilePath, MAX_LOG_BYTES);
      appendFileSync(logFilePath, `${entry}\n`, "utf-8");
    },
  );

  return Logger.add(fileLogger);
};
```
