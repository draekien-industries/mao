# Phase 04.3: Multi-tab orchestration with background streaming - Research

**Researched:** 2026-03-28
**Domain:** Reactive atom state management, concurrent streaming, scroll behavior
**Confidence:** HIGH

## Summary

This phase extends the existing effect-atom architecture (built in Phase 04.1) to support multiple concurrent CLI streams running in background across tabs, with status indicators, smart scroll, and concurrency warnings. The foundation is solid: `sendMessageAtom` already forks independent fibers per call via `appRuntime.fn`, and `Atom.family` keyed by tab ID already isolates per-tab state. The core work is adding new atoms (unread, tool-input, scroll position, auto-scroll, draft input), extending the `tabStatusAtom` derivation from 3 to 5 states (adding unread and tool-input), wiring scroll behavior to the chat panel, and adding a concurrency warning banner to the sidebar.

The key technical insight is that background streaming already works out of the box. Effect fibers spawned by `appRuntime.fn` continue running regardless of which React components are mounted. Atoms with `Atom.keepAlive` retain state across tab switches. The work is primarily about surfacing this concurrent state to the user through indicators, managing scroll position per tab, and detecting the tool-input state from the event stream.

**Primary recommendation:** Build incrementally -- new per-tab atoms first, then extend tabStatusAtom derivation, then wire scroll behavior, then sidebar indicators and concurrency banner. No new libraries needed.

### Pattern 1: Extended TabStatus Derivation

**What:** Extend `tabStatusAtom` from 3-state to 5-state with priority ordering.
**When to use:** Status indicator rendering in sidebar session entries.

The current `tabStatusAtom` derives from `isStreamingAtom` and `errorAtom`. Extend it to read four atoms and apply priority: error > tool-input > unread > streaming > idle.

```typescript
// Priority: error > tool-input > unread > streaming > idle
export type TabStatus = "streaming" | "unread" | "error" | "tool-input" | "idle";

export const tabStatusAtom = Atom.family((tabId: string) =>
  Atom.make((get) => {
    const err = get(errorAtom(tabId));
    const toolInput = get(toolInputAtom(tabId));
    const unread = get(unreadAtom(tabId));
    const streaming = get(isStreamingAtom(tabId));
    if (err !== null) return "error" as const;
    if (toolInput) return "tool-input" as const;
    if (unread) return "unread" as const;
    if (streaming) return "streaming" as const;
    return "idle" as const;
  }),
);
```

### Pattern 2: Tool-Input Detection from Event Stream

**What:** Detect when Claude is waiting for tool confirmation by watching `content_block_start` events with `tool_use` content blocks.
**When to use:** Inside `sendMessageAtom`'s `Stream.runForEach` event handler.

The Claude CLI emits `StreamEventMessage` events. When a `content_block_start` event has a `content_block` of type `tool_use`, Claude is about to execute a tool. The tool-input state should be set when the CLI pauses for user confirmation. The key signal is: an `AssistantMessageEvent` arrives whose content contains a `ToolUseBlock` but no subsequent `ResultEvent` arrives promptly -- the CLI is waiting for tool approval.

However, looking at the actual event flow more carefully: the Claude CLI in `--output-format stream-json` mode emits a `tool_use` top-level event (caught by `UnknownEvent` currently) when waiting for input. The simpler detection approach is:

```typescript
// In sendMessageAtom's Stream.runForEach handler:
if (isAssistantMessage(event)) {
  const hasToolUse = event.message.content.some(
    (block) => block.type === "tool_use"
  );
  if (hasToolUse) {
    ctx.set(toolInputAtom(tabId), true);
  }
  // ... existing text extraction logic
}
```

Tool-input clears when: the stream emits the next event after tool approval (any subsequent event resets it), or when a `ResultEvent` arrives.

### Pattern 4: Smart Scroll with Position Preservation

**What:** Track scroll position per tab, auto-scroll when at bottom, preserve position on tab switch.
**When to use:** Chat panel component.

Implementation approach using refs and atoms:
- `scrollPositionAtom` (Atom.family, number) -- saves scrollTop when switching away from a tab
- `autoScrollAtom` (Atom.family, boolean, default true) -- tracks whether auto-scroll is active
- On scroll event: check if user is "at bottom" (within threshold), update autoScrollAtom accordingly
- On tab switch away: save current scrollTop to scrollPositionAtom
- On tab switch to: restore scrollTop from scrollPositionAtom
- When auto-scroll is ON and new content arrives: scroll to bottom
- When scrolling to bottom: clear unread for that tab

```typescript
// "At bottom" threshold -- typically 50-100px tolerance
const AT_BOTTOM_THRESHOLD = 50;

const isAtBottom = (el: HTMLElement): boolean =>
  el.scrollHeight - el.scrollTop - el.clientHeight < AT_BOTTOM_THRESHOLD;
```

### Pattern 5: Concurrency Tracking Derived Atom

**What:** Count how many tabs are currently streaming.
**When to use:** Sidebar banner display logic.

This needs a derived atom that aggregates across all tab IDs. Since `Atom.family` does not expose an enumeration of active keys, the concurrency count should be tracked via a separate global atom that `sendMessageAtom` increments/decrements.

```typescript
// Global counter, not a family
export const activeStreamCountAtom = Atom.make(0).pipe(Atom.keepAlive);

// In sendMessageAtom: increment on start, decrement on finish/error
ctx.set(activeStreamCountAtom, ctx(activeStreamCountAtom) + 1);
// ... in finally/catchAll:
ctx.set(activeStreamCountAtom, ctx(activeStreamCountAtom) - 1);
```

### Anti-Patterns to Avoid

- **Pausing/resuming streams on tab switch:** Streams must continue running; atoms update regardless of mounted components (D-01, D-02)
- **Clearing unread on tab click:** Unread must only clear on scroll-to-bottom (D-08)
- **Using useCallback/useMemo:** React Compiler handles this (CLAUDE.md rule)
- **Using `as` type casting:** Prefer type narrowing or Schema.is guards (CLAUDE.md rule)

## Don't Hand-Roll

- **Scroll position tracking:** Use native `scrollTop`/`scrollHeight` DOM properties via refs -- no virtual scroll library needed for this use case
- **Debouncing scroll events:** Use `requestAnimationFrame` for scroll handler throttling (already established in codebase via `scrollRafRef` pattern)
- **Concurrency counting:** Simple increment/decrement atom, not a complex semaphore

## Common Pitfalls

### Pitfall 1: Atom.family Key Type Mismatch
**What goes wrong:** Tab IDs are numbers from SQLite but atom family keys are strings. Inconsistent conversion causes duplicate atom instances.
**Why it happens:** `activeTabIdAtom` holds `number | null`, but `messagesAtom` takes `string`.
**How to avoid:** Always use `String(tabId)` consistently. The existing code already does this (`const tabKey = String(session.id)`), maintain this pattern.
**Warning signs:** Status indicator not updating, messages appearing in wrong tab.

### Pitfall 2: Race Condition in Active Stream Count
**What goes wrong:** If `sendMessageAtom` errors before the increment or the decrement runs in the wrong order, the count drifts.
**Why it happens:** Increment/decrement not properly paired with try/finally-style guarantees.
**How to avoid:** Use `Effect.ensuring` or structure the increment before the stream starts and decrement in both the success path and `Effect.catchAll`.
**Warning signs:** Concurrency banner stuck showing even when no streams are active.

### Pitfall 3: Scroll Position Saved Too Late
**What goes wrong:** User switches tab, but scroll position was not yet saved because the save runs in a React effect.
**Why it happens:** React effect cleanup runs asynchronously; by the time it fires, the DOM has already unmounted the scroll container.
**How to avoid:** Save scroll position synchronously in the `setActiveTabAtom` action before changing `activeTabIdAtom`, or save on every scroll event (debounced) into the atom.
**Warning signs:** Tab always scrolls to top or bottom on switch-back instead of preserving position.

### Pitfall 4: Stale Closure in Scroll Event Handler
**What goes wrong:** Scroll handler captures old `tabKey` or atom references after tab switch.
**Why it happens:** Event listener attached with one closure, tab changes, handler still references old values.
**How to avoid:** Re-attach scroll listener when active tab changes, or use a ref that tracks current tab ID.
**Warning signs:** Unread clears on wrong tab, scroll position saved to wrong tab atom.

### Pitfall 5: Tool-Input State Not Clearing
**What goes wrong:** Orange dot persists after tool execution completes.
**Why it happens:** Tool-input was set but never cleared when the stream resumes.
**How to avoid:** Clear `toolInputAtom` on: next `SystemInitEvent`, next `StreamEventMessage`, or `ResultEvent`. Also clear on `sendMessageAtom` start (same as error clearing).
**Warning signs:** Orange dot showing on tab after conversation has continued.

### Pitfall 6: Multiple sendMessageAtom Calls on Same Tab
**What goes wrong:** User sends a message while one is still streaming on the same tab, causing interleaved state updates.
**Why it happens:** Nothing currently prevents double-send on the same tab.
**How to avoid:** The existing `isStreamingAtom` guard (input is disabled while streaming) prevents this in the UI. Ensure this remains enforced.
**Warning signs:** Garbled messages, duplicate entries in messages array.

## State of the Art

- effect-atom's `Atom.keepAlive` is the key enabler -- atoms retain values even when no component subscribes, so background tab state persists naturally
- `appRuntime.fn` forks Effect fibers that run independently of React component lifecycle -- this is the foundation for background streaming
- The `FnContext` interface provides `ctx()` for reading and `ctx.set()` for writing atoms synchronously within Effect programs, enabling event-driven state updates from streams

## Open Questions

1. **Tool-input detection accuracy**
   - What we know: `AssistantMessageEvent` with `ToolUseBlock` content indicates tool use. The CLI may also emit a top-level `tool_use` event (currently caught by `UnknownEvent`).
   - What's unclear: Exact event that signals "waiting for tool approval" vs "tool already executed". In non-interactive `--output-format stream-json` mode, the CLI may auto-approve tools.
   - Recommendation: Start with detecting `ToolUseBlock` in `AssistantMessageEvent.message.content` as the trigger. If this is inaccurate, refine based on testing. Clear on any subsequent event.

2. **Batch rendering vs live-resume on tab switch**
   - What we know: Atoms update in background; when component re-mounts, `useAtomValue` reads current state immediately.
   - What's unclear: Whether accumulated streaming text appears all at once or streams in naturally.
   - Recommendation: effect-atom provides the current value on mount -- so the user sees all accumulated content immediately (batch). This is the correct behavior; no special handling needed.

3. **Warning color token**
   - What we know: shadcn semantic colors include `destructive` but may not include `warning` by default.
   - What's unclear: Whether the project has a warning color token configured.
   - Recommendation: Check `src/index.css` for existing warning tokens. If absent, add `--warning` and `--warning-foreground` CSS custom properties, or use a neutral alternative like `bg-amber-500/10 text-amber-700`.
