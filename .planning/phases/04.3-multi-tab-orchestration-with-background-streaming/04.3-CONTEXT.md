# Phase 04.3: Multi-tab orchestration with background streaming - Context

**Gathered:** 2026-03-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Enable multiple concurrent CLI streams running in background across tabs. When a user switches tabs, active streams continue and atom state updates regardless of which tab is visible. Status indicators (streaming/unread/error) reflect real-time state in the sidebar. A soft concurrency limit warns users when too many streams are active. Does not include tab restore from persistence or graceful shutdown — that's Phase 5.

</domain>

<decisions>
## Implementation Decisions

### Background streaming
- **D-01:** Streams continue running in the background when the user switches tabs. The Effect program keeps running, atom state keeps updating. When the user switches back, they see all accumulated text.
- **D-02:** This is enabled by the centralized effect-atom state from Phase 04.1 — atoms update regardless of which component is mounted. No special pause/resume logic needed.

### Status indicators
- **D-03:** Three status indicators on session entries in the sidebar:
  - **Streaming pulse** — a pulsing dot when Claude is actively responding
  - **Unread badge** — highlighted when a tab received a response the user hasn't viewed yet (switched away while streaming)
  - **Error indicator** — distinct visual (red dot/icon) when a session hit an error the user hasn't seen
- **D-04:** Status indicators derive from atom state — the streaming/idle/error `Result` type from effect-atom drives the indicators directly.
- **D-05:** "Unread" state clears when the user switches to (views) the tab.

### Concurrency management
- **D-06:** Soft limit with warning — allow unlimited concurrent streams, but warn the user after N concurrent streams (exact threshold TBD during planning) that performance may degrade.
- **D-07:** The warning is non-blocking — users can dismiss it and continue. No hard cap.

### Claude's Discretion
- Exact concurrency warning threshold (3, 5, or dynamic based on system resources)
- Warning UI design (toast, inline banner, sidebar indicator)
- How "unread" state is tracked (atom flag, timestamp comparison, etc.)
- Whether the streaming indicator animates differently based on stream progress
- Performance optimizations for many concurrent atom updates
- How error state interacts with unread state (error supersedes unread, or both shown)

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- effect-atom `Result` type (from Phase 04.1) — `Result.initial`, `Result.success`, `Result.failure` map directly to the three status indicators
- `Atom.family` per-tab atoms (from Phase 04.1) — each tab's atom holds its own streaming state, enabling independent concurrent streams
- `Atom.pull(stream)` (from Phase 04.1) — CLI streams feed into atoms, streams naturally run in background via Effect fibers

### Established Patterns
- **Fiber management**: `runtime.runFork` in current `useClaudeChat` — replaced by atom-managed fibers, but same principle: fibers run independently
- **Error handling**: `Effect.catchAll` captures errors into state — same pattern with atom Result type
- **Stream consumption**: `Stream.runForEach` with per-event dispatch — replaced by `Atom.pull` but event handling logic transfers

### Integration Points
- Sidebar session entries — status indicator components need atom subscriptions via `useAtomValue`
- Tab switching logic — "unread" flag atom set when stream completes on non-visible tab, cleared on tab focus
- Concurrency tracking — a derived atom that counts active streaming atoms, triggers warning when threshold exceeded

</code_context>

<specifics>
## Specific Ideas

- Status indicators driven directly by effect-atom Result type — no separate status tracking mechanism needed
- Unread badge clears on tab focus, not on tab click (user must actually see the content)
- Concurrency warning as a non-blocking dismissible notification

</specifics>

<deferred>
## Deferred Ideas

- Hard concurrency cap with queuing — not needed for v1, soft warning is sufficient
- Per-project concurrency limits — all sessions share the same global limit
- Stream priority/ordering — all streams are equal, no prioritization
- Resource monitoring (CPU/memory) for dynamic concurrency limits — defer to future iteration

</deferred>
