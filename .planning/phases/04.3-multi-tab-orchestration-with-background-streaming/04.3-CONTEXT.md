# Phase 04.3: Multi-tab orchestration with background streaming - Context

**Gathered:** 2026-03-28 (updated)
**Status:** Ready for planning

<domain>
## Phase Boundary

Enable multiple concurrent CLI streams running in background across tabs. When a user switches tabs, active streams continue and atom state updates regardless of which tab is visible. Status indicators (streaming/unread/error/tool-input) reflect real-time state in the sidebar. A soft concurrency limit warns users when too many streams are active via an inline sidebar banner. Smart scroll preserves per-tab scroll position and controls auto-scroll behavior. Does not include tab restore from persistence or graceful shutdown — that's Phase 5.

</domain>

<decisions>
## Implementation Decisions

### Background streaming
- **D-01:** Streams continue running in the background when the user switches tabs. The Effect program keeps running, atom state keeps updating. When the user switches back, they see all accumulated text.
- **D-02:** This is enabled by the centralized effect-atom state from Phase 04.1 — atoms update regardless of which component is mounted. No special pause/resume logic needed. `sendMessageAtom` is already a single global action (not a family), so fibers are never tied to per-tab subscriptions.

### Status indicators
- **D-03:** Four status indicators on session entries in the sidebar, displayed as a small colored dot on the right edge of the session entry:
  - **Streaming** — pulsing gray dot when Claude is actively responding
  - **Unread** — solid green dot when a tab received a completed response the user hasn't scrolled to yet
  - **Error** — solid red dot when a session hit an error the user hasn't retried
  - **Tool input** — solid orange dot when Claude is waiting for tool confirmation/approval from the user
- **D-04:** Status indicators derive from atom state — the streaming/idle/error state drives the indicators, plus a new `toolInputAtom` for the orange state and `unreadAtom` for the green state.
- **D-05:** Error supersedes unread — when both states are true, only the error indicator shows.
- **D-06:** No indicator shown for idle state (no dot visible).

### Unread tracking
- **D-07:** Tab marked unread only when a full assistant response finishes (AssistantMessageEvent) while the tab is not the active tab. Partial streaming content does not trigger unread.
- **D-08:** Unread clears when the user scrolls to the bottom of the chat in that tab — not on tab click. This ties into the smart scroll behavior.
- **D-09:** Error indicator persists until the user sends a new message or explicitly retries. It does NOT clear on tab view/click.

### Concurrency management
- **D-10:** Soft limit with warning at 5 concurrent streams. Allow unlimited concurrent streams, but display a warning after 5 concurrent streams that performance may degrade.
- **D-11:** The warning is an inline banner at the top of the sidebar. Non-blocking — users can continue working.
- **D-12:** The banner auto-dismisses when concurrent stream count drops below 5. Reactive to current state.

### Tab switch behavior
- **D-13:** Each tab preserves its scroll position independently. Switching back to a tab puts the user exactly where they left off.
- **D-14:** Smart auto-scroll: auto-scroll is ON when user is at the bottom of the chat. If user scrolls up from the bottom, auto-scroll disables. Scrolling back to the bottom re-enables auto-scroll and clears unread.
- **D-15:** Instant swap on tab switch — no animation or transition between tab contents.
- **D-16:** Each tab preserves its input field draft text when switching away. Per-tab input atom via Atom.family.

### Claude's Discretion
- Whether batch-rendering accumulated content or live-resuming streaming on tab switch back (depends on what effect-atom naturally provides)
- `TabStatus` type design — how to extend the current three-state union to four states with tool-input detection
- How tool-input state is detected from the event stream (which event signals Claude is waiting for tool confirmation)
- Per-tab scroll position tracking implementation (refs, atoms, or DOM state)
- Performance optimizations for many concurrent atom updates
- Smart scroll threshold for "at bottom" detection (exact pixel tolerance)
- Concurrency count implementation — derived atom aggregating all active tab IDs

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `sendMessageAtom` (global action, not family) — Already forks independent fibers per call, supports concurrent streams out of the box
- `tabStatusAtom` family — Currently derives `"streaming" | "error" | "idle"` from `isStreamingAtom` and `errorAtom`. Needs extension to four states (add unread + tool-input)
- `Atom.family` per-tab atoms — Each tab's atom holds its own streaming state, enabling independent concurrent streams
- `activeTabIdAtom` — Tracks which tab is visible, used to determine "non-visible" for unread detection
- `setActiveTabAtom` — Current tab switch action, integration point for scroll preservation

### Established Patterns
- **Fiber management**: `appRuntime.fn` in `sendMessageAtom` — forks fibers independently, each tab's stream runs in its own fiber
- **Error handling**: `Effect.catchAll` captures errors into `errorAtom` per tab
- **Stream event processing**: `Stream.runForEach` with per-event dispatch in `sendMessageAtom` — extend this to detect tool confirmation events
- **Atom state derivation**: `tabStatusAtom` uses `Atom.make((get) => ...)` computed pattern

### Integration Points
- `sendMessageAtom` event loop — Add unread flag setting when AssistantMessageEvent arrives on non-active tab
- `tabStatusAtom` derivation — Extend from 3-state to 4-state (add unread + tool-input atoms as inputs)
- `setActiveTabAtom` — No longer clears unread (that's tied to scroll position now)
- Sidebar session entries — Status indicator dot component subscribes to extended `tabStatusAtom`
- New atoms needed: `unreadAtom` (per-tab boolean), `toolInputAtom` (per-tab boolean), `scrollPositionAtom` (per-tab number), `autoScrollAtom` (per-tab boolean), `draftInputAtom` (per-tab string)
- Concurrency tracking — New derived atom counting tabs where `isStreamingAtom` is true

</code_context>

<specifics>
## Specific Ideas

- Status indicators as right-side dots on session entries: gray pulsing (streaming), green solid (unread), red solid (error), orange solid (tool input)
- Error supersedes unread — only one dot shown at a time with priority: error > tool-input > unread > streaming
- Unread clears by scrolling to bottom, NOT by tab click — ties user acknowledgment to actually seeing the content
- Smart scroll: auto-scroll when at bottom, disable on scroll up, re-enable on scroll to bottom
- Per-tab draft text preservation via `draftInputAtom` family
- Inline sidebar banner for concurrency warning (5+ streams), auto-dismisses when count drops

</specifics>

<deferred>
## Deferred Ideas

- Hard concurrency cap with queuing — not needed for v1, soft warning is sufficient
- Per-project concurrency limits — all sessions share the same global limit
- Stream priority/ordering — all streams are equal, no prioritization
- Resource monitoring (CPU/memory) for dynamic concurrency limits — defer to future iteration
- Tab transition animations — decided against for now (instant swap)

</deferred>
