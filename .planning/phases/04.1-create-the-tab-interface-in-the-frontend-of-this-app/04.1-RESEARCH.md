# Phase 04.1: Atom state foundation + sidebar shell - Research

**Researched:** 2026-03-27
**Domain:** effect-atom state management + shadcn sidebar composition
**Confidence:** MEDIUM

## Summary

This phase installs `@effect-atom/atom-react` (which bundles `@effect-atom/atom` as a dependency) and migrates the single-session chat from the `useClaudeChat` hook's internal `useState`/`useRef` pattern to atom-based reactive state. It also builds the two-tiered sidebar component using the existing shadcn sidebar primitives plus a new `collapsible` component.

The core technical challenge is replacing `ManagedRuntime` + React context (`RuntimeProvider`) with `Atom.runtime(layer)` + `RegistryProvider`. The `Atom.make` function accepts Effect streams directly and returns `Atom<Result<A, E>>`, which maps naturally to the streaming/idle/error states the sidebar status indicators need. However, `Atom.pull` is designed for paginated/on-demand consumption (not continuous event streams) -- the correct pattern for consuming CLI streams is `Atom.make((get) => stream)` which auto-subscribes and emits the latest value via `Result`.

**Primary recommendation:** Install `@effect-atom/atom-react` (v0.5.0) with `@effect/experimental` (v0.58.0+) as new dependencies. Use `Atom.runtime(layer)` as the single point of service injection, replacing the current `AppRuntime` + `RuntimeProvider`. Use `Atom.family` keyed by tab ID for per-session state. Use `Atom.fn` for the `sendMessage` action.

### Pattern 1: Atom Runtime (replaces ManagedRuntime + React Context)

**What:** `Atom.runtime(layer)` creates an `AtomRuntime` that provides Effect services to atoms. This replaces the current `ManagedRuntime.make(layer)` + `RuntimeProvider` pattern.

**Current code (to be replaced):**
```typescript
// src/services/claude-rpc/runtime.ts
export const AppRuntime = ManagedRuntime.make(ClaudeCliFromRpc);
const RuntimeContext = createContext<AppRuntimeType>(AppRuntime);
export const RuntimeProvider = RuntimeContext.Provider;
export const useRuntime = () => useContext(RuntimeContext);
```

**New pattern:**
```typescript
// src/atoms/runtime.ts
import { Atom } from "@effect-atom/atom-react"
import { ClaudeCliFromRpc } from "@/services/claude-rpc/client"

export const appRuntime = Atom.runtime(ClaudeCliFromRpc)
```

**Provider swap in app.tsx:**
```typescript
// Before:
<RuntimeProvider value={AppRuntime}>
  <RouterProvider router={router} />
</RuntimeProvider>

// After:
import { RegistryProvider } from "@effect-atom/atom-react"
<RegistryProvider>
  <RouterProvider router={router} />
</RegistryProvider>
```

The `RegistryProvider` manages atom lifecycle (creation, cleanup, garbage collection). No explicit runtime value needs to be passed -- atoms access services through the `appRuntime` atom.

### Pattern 2: Per-Tab Chat State via Atom.family

**What:** `Atom.family` creates a stable atom reference per key. Each tab gets its own set of state atoms keyed by tab ID.

```typescript
// src/atoms/chat.ts
import { Atom, Result } from "@effect-atom/atom-react"
import type { ChatMessage } from "@/services/database/session-reconstructor/schemas"

// Simple writable atoms per tab
export const messagesAtom = Atom.family((tabId: string) =>
  Atom.make<ReadonlyArray<ChatMessage>>([]).pipe(Atom.keepAlive)
)

export const streamingTextAtom = Atom.family((tabId: string) =>
  Atom.make("").pipe(Atom.keepAlive)
)

export const sessionIdAtom = Atom.family((tabId: string) =>
  Atom.make<string | null>(null).pipe(Atom.keepAlive)
)

// Derived status atom for sidebar indicators
export const tabStatusAtom = Atom.family((tabId: string) =>
  Atom.make((get) => {
    const streaming = get(streamingTextAtom(tabId))
    // Derive status from streaming state
    return streaming !== "" ? "streaming" as const : "idle" as const
  })
)
```

`Atom.keepAlive` prevents atoms from being garbage collected when no component subscribes. This is critical for background tabs -- their state must persist even when the tab's chat panel is not rendered.

### Pattern 3: Stream Consumption via Atom.make with Stream

**What:** `Atom.make` accepts a `Stream.Stream` and returns `Atom<Result<A, E>>`. The stream is subscribed when the atom is first read and cleaned up when the atom is disposed.

**IMPORTANT CORRECTION to D-04:** The CONTEXT.md specifies `Atom.pull(stream)` for CLI stream consumption. However, research reveals `Atom.pull` is designed for **paginated/on-demand** consumption (it accumulates items and requires explicit pulls). For continuous event streams like the Claude CLI output, the correct approach is `Atom.make((get) => stream)` which auto-subscribes and delivers each emitted value through the `Result` type. Alternatively, use `Atom.fn` to create a callable action that runs an Effect program imperatively -- this is closer to the current `runtime.runFork` pattern.

**Recommended approach using Atom.fn:**
```typescript
// src/atoms/chat.ts
import { Atom } from "@effect-atom/atom-react"
import { Effect, Stream } from "effect"
import { appRuntime } from "./runtime"

export const sendMessageAtom = Atom.family((tabId: string) =>
  appRuntime.fn(
    Effect.fnUntraced(function* (prompt: string) {
      const cli = yield* ClaudeCli
      const sessionId = /* read from sessionIdAtom */
      const stream = sessionId
        ? cli.resume(new ResumeParams({ prompt, session_id: sessionId }))
        : cli.query(new QueryParams({ prompt }))

      yield* Stream.runForEach(stream, (event) =>
        Effect.sync(() => {
          // Update messagesAtom, streamingTextAtom, sessionIdAtom
          // via atom set functions passed through closure
        })
      )
    })
  )
)
```

In the component:
```typescript
const sendMessage = useAtomSet(sendMessageAtom(tabId), { mode: "promiseExit" })
// sendMessage("hello") returns Promise<Exit<void, Error>>
```

### Anti-Patterns to Avoid

- **Using Atom.pull for continuous streams:** `Atom.pull` requires explicit pull invocations and accumulates items. It is for pagination, not event stream consumption.
- **Forgetting Atom.keepAlive on tab state:** Without `keepAlive`, atoms reset when no component reads them. Background tab state would be lost.
- **Creating atoms inside components:** Atoms must be created at module level or via `Atom.family`. Creating inside components causes new atom identity on every render.
- **Using `as` casting for the IPC boundary:** The existing `message as FromServerEncoded` in client.ts is an inherited pattern. New code should prefer Schema decoding per CLAUDE.md rules.
- **Using useCallback/useMemo in new components:** Per CLAUDE.md, depend on the React Compiler instead.

## Don't Hand-Roll

- **State management plumbing:** Do not build a custom React Context + useReducer system for tab state. Use effect-atom's `Atom.family` + `RegistryProvider`.
- **Stream-to-state bridging:** Do not create manual `useEffect` + `runtime.runFork` patterns. Use `Atom.fn` or `Atom.make` with streams.
- **Collapsible groups:** Do not build custom expand/collapse with state + CSS transitions. Use shadcn's `Collapsible` component.
- **Sidebar layout and mobile responsiveness:** Do not hand-roll a sidebar. The shadcn sidebar handles desktop/mobile breakpoints, keyboard shortcut (Ctrl+B), and sheet overlay automatically.

## Common Pitfalls

### Pitfall 1: Atom.pull vs continuous stream consumption
**What goes wrong:** Using `Atom.pull(cliStream)` for the Claude CLI stream results in items only being consumed when explicitly "pulled" -- the stream does not run continuously.
**Why it happens:** `Atom.pull` is designed for paginated data (infinite scroll). The CONTEXT.md decision D-04 mentions `Atom.pull` but this is a misunderstanding of the API.
**How to avoid:** Use `Atom.fn` with `Effect.fnUntraced` to create an imperative action that runs `Stream.runForEach` and updates writable atoms as events arrive. This is the closest equivalent to the current `runtime.runFork` pattern.
**Warning signs:** Stream only emits one batch and stops; `PullResult.done` is true after first pull.

### Pitfall 2: React 19 peer dependency warning
**What goes wrong:** `npm install` fails or warns because `@effect-atom/atom-react` declares `react: >=18 <20`.
**Why it happens:** The library hasn't updated its peer dep range for React 19 yet.
**How to avoid:** Use `npm install --legacy-peer-deps` or add an `overrides` field in package.json: `"overrides": { "@effect-atom/atom-react": { "react": "$react" } }`.
**Warning signs:** npm ERR! ERESOLVE during install.

### Pitfall 3: @effect/experimental pulling in unwanted dependencies
**What goes wrong:** `@effect/experimental` has optional peer deps on `lmdb` and `ioredis` which are large native modules.
**Why it happens:** These are listed as peer deps (not dependencies) so they are optional, but npm may warn.
**How to avoid:** They are optional peers and won't be installed automatically. Ignore the warnings. No action needed.

### Pitfall 4: Atom cleanup on tab close
**What goes wrong:** Atoms created via `Atom.family` persist indefinitely with `keepAlive`, causing memory leaks for closed tabs.
**Why it happens:** `Atom.keepAlive` prevents GC even when no subscribers exist.
**How to avoid:** For this phase (single tab, mock data), this is not a concern. In Phase 04.2+, implement explicit cleanup when tabs are closed. Effect-atom supports finalizers via `Effect.addFinalizer` in atom constructors.
**Warning signs:** Growing memory usage over time in production.

### Pitfall 5: SidebarProvider min-h-svh breaking Electron layout
**What goes wrong:** The shadcn `SidebarProvider` applies `min-h-svh` (min-height: 100svh) which may not work correctly in Electron's BrowserWindow.
**Why it happens:** `svh` is a viewport unit designed for mobile browsers with dynamic toolbars. Electron windows don't have this behavior.
**How to avoid:** Override with `className="min-h-screen"` or `className="h-screen"` on `SidebarProvider`. The existing root layout already uses `h-screen`.
**Warning signs:** Content overflows or scrollbar appears at the app level.

## State of the Art

- **effect-rx renamed to effect-atom:** The `@effect-rx` package was renamed to `@effect-atom` in early 2025. The npm packages are `@effect-atom/atom` and `@effect-atom/atom-react`. References to `effect-rx` or `@effect-rx/rx-react` are outdated.
- **AtomRpc module:** A new module integrating `@effect/rpc` with effect-atom was added alongside the rename. This could potentially simplify the RPC client in future phases but is not needed for 04.1.
- **React Compiler compatibility:** effect-atom's hooks use `useSyncExternalStore` which is compatible with React Compiler. No special handling needed.

## Open Questions

1. **Atom.pull vs Atom.make for stream consumption**
   - What we know: `Atom.pull` is for paginated data, `Atom.make` with streams auto-subscribes, and `Atom.fn` allows imperative Effect execution
   - What's unclear: CONTEXT.md D-04 specifies `Atom.pull` but this seems wrong. The planner should use `Atom.fn` pattern instead.
   - Recommendation: Use `Atom.fn` with `Stream.runForEach` to update writable atoms. This most closely mirrors the existing `runtime.runFork` pattern and gives full control over event handling.

2. **ClaudeCliFromRpc layer compatibility with Atom.runtime**
   - What we know: `ClaudeCliFromRpc` is a scoped layer (uses `Layer.scoped`). `Atom.runtime` should handle scoped layers.
   - What's unclear: Whether `Atom.runtime` manages the scope lifecycle correctly for long-lived RPC connections.
   - Recommendation: Test early. If scoped layer is problematic, extract the transport setup to a non-scoped layer.

3. **Removing ManagedRuntime entirely**
   - What we know: The renderer currently uses `AppRuntime = ManagedRuntime.make(ClaudeCliFromRpc)` which the `useClaudeChat` hook calls via `runtime.runFork`.
   - What's unclear: Whether `Atom.runtime` fully replaces all `ManagedRuntime` use cases, or if some escape hatch is still needed.
   - Recommendation: Plan for full replacement. `Atom.fn` + `useAtomSet` with `mode: "promiseExit"` replaces `runtime.runFork`.
