# Phase 04.1: Atom state foundation + sidebar shell - Context

**Gathered:** 2026-03-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Install the `effect-atom` library and migrate the single-session chat from `useClaudeChat` hook state to atom-based reactive state. Build the two-tiered sidebar component (projects as collapsible group headers, sessions nested underneath) with basic structure. Proves the new architecture works end-to-end with one tab. Does not include project registration, session creation flows, git operations, or multi-stream orchestration — those are Phases 04.2 and 04.3.

</domain>

<decisions>
## Implementation Decisions

### State management library
- **D-01:** Use `effect-atom` (https://github.com/tim-smart/effect-atom) as the centralized state management layer for tabs. Replaces the current `useClaudeChat` hook's internal `useState`/`useRef` pattern with atom-based reactive state.
- **D-02:** Use `Atom.runtime(layer)` to integrate with the existing Effect layer system, replacing the current `AppRuntime` + `RuntimeProvider` pattern in `app.tsx`.
- **D-03:** Use `Atom.family(fn)` to create per-tab state atoms keyed by tab ID. Each tab's state (messages, streaming text, session ID, status) lives in its own atom.
- **D-04:** Use `Atom.pull(stream)` for CLI stream consumption, replacing the current `runtime.runFork` + `setState` fire-and-forget pattern.
- **D-05:** Use the `Result` type from effect-atom for streaming/idle/error states, which also drives status indicators in the sidebar.

### Sidebar layout
- **D-06:** Left sidebar using the existing `sidebar` shadcn component. Two-tiered structure: projects as collapsible group headers, sessions nested underneath.
- **D-07:** Sidebar content area is scrollable. Project groups are collapsible.
- **D-08:** Session entries show: branch/worktree label + status indicators (streaming pulse, unread badge, error indicator).

### Scope for this phase
- **D-09:** This phase delivers the sidebar shell with hardcoded/mock project and session entries, plus one working chat session running through the atom architecture. The goal is to prove effect-atom works end-to-end before adding project/session management.

### Claude's Discretion
- Atom structure design (which atoms, how they compose, naming)
- How to bridge effect-atom's React hooks with the existing component structure
- Sidebar component composition (which shadcn sidebar sub-components to use)
- How status indicator atoms derive from the streaming state
- Migration strategy for `useClaudeChat` — incremental or full rewrite

</decisions>

## Deferred Ideas

### Phase 04.2 (decided during this discussion)

- Immediate project creation with first session auto-created on current branch

- Atom state updates regardless of visible tab
- Soft concurrency limit with warning after N concurrent streams
- Unread indicator when tab receives response while not visible

- Session hydration in the UI
- Graceful shutdown ensuring data safety
